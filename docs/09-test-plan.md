# План тестирования — LegalFlow

## Цель

Обеспечить стабильность MVP перед запуском пилота: корректная работа аутентификации, CRUD-операций, AI-функций, уведомлений и UI.

## Уровни тестирования

| Уровень | Что проверяем | Инструменты | Периодичность |
|---|---|---|---|
| **Smoke-тесты** | Критичные сценарии: логин, создание дела, загрузка документа | Ручные + Playwright | После каждого деплоя |
| **Функциональное тестирование** | User stories, формы, валидация, RBAC | Playwright + ручные | Перед каждым релизом |
| **Интеграционное тестирование** | API Clerk, OpenAI/OpenRouter, Telegram Bot API | Postman / pytest | Перед релизом и при изменении интеграций |
| **Нагрузочное тестирование** | Одновременные AI-запросы, загрузка документов | Locust / k6 | Перед публичным запуском |
| **Регрессионное тестирование** | Ранее работавшие фичи не сломались | Playwright + чек-лист | Перед каждым релизом |

## Инструменты

- **Playwright** — end-to-end тесты основных пользовательских сценариев.
- **Postman / Newman** — тестирование REST API и интеграций.
- **pytest** — unit- и интеграционные тесты бэкенда (если добавляются).
- **Locust / k6** — нагрузочное тестирование.
- **Allure** — отчёты о тестировании (опционально).

## Тест-кейсы

### TC-001 — Создание и анализ документа

**Шаги:**
1. Авторизоваться в системе.
2. Перейти в раздел «Документы».
3. Нажать «Загрузить документ» и выбрать файл `.docx`.
4. Дождаться появления документа в списке.
5. Открыть документ и нажать «Анализировать».

**Ожидаемый результат:**
- Файл успешно загружен и отображается.
- AI-анализ возвращает структурированный результат (риски, обязательства, сроки).
- Время анализа ≤ 10 секунд.
- Результат сохраняется в истории.

### TC-002 — Создание задачи и уведомление в Telegram

**Шаги:**
1. Авторизоваться и подключить Telegram в настройках.
2. Создать новую задачу с дедлайном «завтра».
3. Назначить себя ответственным.
4. Дождаться срабатывания напоминания.

**Ожидаемый результат:**
- Задача отображается в списке / на Kanban-доске.
- В Telegram приходит сообщение с текстом задачи и ссылкой.
- Ссылка открывает LegalFlow с выбранной задачей.

### TC-003 — Глобальный поиск

**Шаги:**
1. Создать клиента, дело и задачу с уникальным названием.
2. Ввести часть названия в строку поиска в шапке на страницах Дашборд, Документы, Задачи.

**Ожидаемый результат:**
- На странице Документы отображаются отфильтрованные документы (серверный поиск).
- На странице Задачи отображаются отфильтрованные задачи (клиентский фильтр).
- Пустой запрос сбрасывает фильтр.

### TC-004 — Ролевая модель

**Шаги:**
1. Войти как owner и создать пользователя с ролью paralegal.
2. Войти под paralegal.
3. Попытаться открыть административные функции / чужие дела.

**Ожидаемый результат:**
- Paralegal видит только разрешённый контент.
- Попытка доступа к закрытым данным возвращает 403 или скрывает UI-элементы.

### TC-005 — Смена языка

**Шаги:**
1. Войти в систему.
2. Перейти в настройки.
3. Сменить язык с русского на английский и наоборот.

**Ожидаемый результат:**
- Интерфейс полностью переключается на выбранный язык.
- Выбор сохраняется после перезагрузки страницы.

### TC-006 — AI-генерация шаблона

**Шаги:**
1. Перейти в раздел «AI-инструменты».
2. Ввести описание дела: «Договор аренды нежилого помещения 50 м², арендатор ООО Ромашка».
3. Выбрать тип документа «Аренда».
4. Нажать «Сгенерировать».

**Ожидаемый результат:**
- Сгенерированный текст содержит обязательные блоки: стороны, объект, срок, арендная плата.
- Текст можно скопировать или сохранить как документ.
- Генерация завершается ≤ 10 секунд.

## Критерии приёмки MVP

- Все smoke-тесты проходят.
- Покрытие функциональных тест-кейсов ≥ 90%.
- Интеграционные тесты с Clerk и Telegram проходят стабильно.
- AI-функции работают с таймаутом ≤ 15 секунд.
- Нет критических и высоких дефектов.

## Что тестируем вручную на пилоте

- Удобство onboarding для нового пользователя.
- Понятность AI-результатов (юристы проверяют качество анализа).
- Корректность Telegram-уведомлений в реальных условиях.
- Восприятие дашборда и воронки дел руководителями.

## Дальнейшее развитие автоматизации

- Добавить Playwright-тесты в CI/Replit workflow.
- Покрыть API тестами на уровне backend-роутов.
- Настроить нагрузочные тесты перед выходом из пилота.
